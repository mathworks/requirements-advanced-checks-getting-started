Managing Requirements

 

Your new design will be based on an existing cruise control design from another vehicle's software, with some minor requirements changes. This is commonly referred to as a "requirements change" workflow.

You can make these modifications by using Simulink Requirements . Simulink Requirements can be used to view and author requirements within Simulink, as well as to establish traceability between requirements, Simulink models, and test cases to understand the impact of a change in requirements.

Let's see how Simulink Requirements is used in a requirements change workflow.

Handling a Requirements Change

Let's use the Requirements Editor to import the changed requirement.

We will start with a baseline set of requirements. The baseline set has already been imported from Microsoft Excel and linked within Simulink Requirements to elements in the design in Stateflow for traceability. We first need to understand what has changed from this baseline.

Step 1: Import the new requirements into Simulink Requirements. Let's first take a look at the requirements, which we've exported from a third party requirements management tool into a Microsoft Exel spreadsheet. " Open the spreadsheet by clicking here.

 

The requirement that has changed has an ID of "INTF-1". The new requirement is the addition of a fourth button called "Cancel", which cancels active cruise control mode.

To import this requirement change in Simulink Requirements, do the following:

  Click here to open the "CruiseControl_Req_ReqChange.slx" Simulink model.

  Import the requirement files in the Requirements Editor

  Let's use the update feature in Simulink Requirements to import the requirement change. Right click on the SystemReqs file and click on Update Source Document name or location

 

 

 

 

  Choose the SystemReqsImport.xlsx file to link the corresponding Excel sheet.

 

  Next click on the Update button to update the requirement file with the new requirement.

 

 

 

  Once the update has completed, notice there is a summary of changes in the Comments section that indicates the "INTF-1" has been updated or changed.

 

 

 

  There is a highlighting feature to show possibly impacted requirements. To highlight the changed requirement, select Display/Change Information.

 

 

 

  Navigate to the changed requirement "INTF-1" to investigate the impact of the change which shows (3) linked requirements in the "Interface" section of the lower level requirements may be impacted.

 

 

 

 

  As described in the tooltip, to clear the issue open the Links view.

 

 

 

The requirements with Change Issues will not require any further handling so we can clear these change issues but we will need to add requirements to include the "Cancel" button in the "Interface" section and add a requirement to describe the functional behavior in the "Functional" section.

 

  In the Links view we can clear the issues by right-clicking at the links file level and select Clear All Change Issues.

 

 

 

To add the (2) new requirements, it is assumed that you can intuitively add these requirements based on the popup menus in the Editor and link the new requirements to the model.

OR

You can load the requirement set changes based on the new "Cancel" button from the "INTF-1" change by loading the MW_CruiseControl_ReqChange and the SystemReqs_ReqChange in the Requirements Editor

 

 

  Next, based on requirements changes as mentioned above, note the (2) new requirements added to the "MW_CruiseControl" requirement set.

 

 

 

 

  We also linked the (2) new requirements to the (2) model modifications required to implement the new requirements. Check these links in the Requirements Perspective.
